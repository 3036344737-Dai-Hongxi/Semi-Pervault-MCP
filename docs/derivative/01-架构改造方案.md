# Pervault 衍生产品 · 架构改造方案 v2

> 目标：把 Neo-Pervault 里「本地优先的个人记忆引擎 + agent 内核」从语音 App 中抽离成一个独立、可复用的 `memory-core`，
> 并以**薄适配层**的方式接入主流软件。第一条产品线优先做 **MCP 记忆服务器**，第二条做 **浏览器扩展**。
>
> 本文是「先出架构改造方案」阶段的交付物，不改任何运行行为，只定义目标结构、决策、分步计划与验收口径。
>
> **v2 变更**：经过第二轮「对抗式」代码核实（写入路径 / 多进程并发 / 降级 / 单用户假设），补齐了 4 个承重洞，并修正了 daemon 与 MCP 的进程模型。逐条变更见 §11。

---

## 1. 背景与定位

现有项目表面是「本地语音 AI 秘书」，但语音录入、转写、前端那一套只是**记忆引擎的一个消费者**。真正可复用、有壁垒的是底下两层：

- **数据层（一个 SQLite 文件）**：四层记忆架构 `memory_items`（情节）→ `structured_facts`（事实）→ `user_persona`（人设）→ `memory_reflection`（洞察），外加轻量知识图谱 `graph_nodes/edges`、向量表 `vec_items`（sqlite-vec）、全文表 `memory_fts`（FTS5）、作业队列 `background_jobs`、调度日志、会话历史。
- **agent 内核（`backend/services/`）**：混合检索 + 意图分流（project/preference/task/persona query）、准入分层、重要度评分、权重衰减、consolidation 整合、sleep agent 睡眠重组、background jobs worker。模型层 OpenAI 兼容、provider 无关。

衍生产品的核心命题：**不是再做一个 App，而是把这颗「个人记忆大脑」做成可被多个宿主共享的内核**——同一份私有记忆，既能喂给 Claude/Cursor（MCP），也能喂给浏览器里的网页助手（扩展）。

## 2. 可行性勘查结论（基于现有代码实测）

> 这一节是动手前对「内核能否干净抽离 + 能否持续跑」的实测，直接决定本方案可信度。

### 2.1 第一轮：内核能否干净抽离

| 勘查项 | 结论 | 证据 |
|---|---|---|
| `services/` 是否依赖 Web 框架 | **几乎不依赖** | 整个 `backend/services/` 只有 **1 行** `from fastapi import HTTPException`（`memory_service.py:20`），其余只 import `database` / `models` / `services.*` / `openai` / `sqlite_vec` / 标准库 |
| 后台 agent 循环是否框架无关 | **是** | `run_periodically`（consolidation）、`run_decay_periodically`、`run_sleep_agent_periodically`、`run_background_jobs_worker` 都是纯 asyncio 协程，`main.py` 只是用 `asyncio.create_task` 把它们挂起来 |
| 数据层是否可独立 | **是** | `database.py` 自带 schema、迁移、`get_db()` / `get_shared_db()`，无 Web 依赖；WAL 模式已开启 |
| 模型层是否 provider 锁定 | **否** | 全部走 OpenAI 兼容 SDK，LLM / Embedding / Whisper 各自独立 env（`OPENAI_*`、`EMBEDDING_*`、`WHISPER_*`），换底模只改 env |
| Web 层是「内核」还是「适配器」 | **是适配器** | `main.py` + `routers/*` + auth cookie 中间件 + rate_limit，本质是「给 Next.js 前端用的 HTTP 宿主」，可与未来的 MCP/浏览器宿主并列 |

**核心结论**：记忆引擎对 FastAPI 的耦合度 ≈ 0（仅 1 处 `HTTPException`，替换成领域异常即可）。抽 `memory-core` 是**打包/重构**工作，不是重写。

### 2.2 第二轮：能否「持续地跑」（对抗式核实）

> 针对「让别人押注 MCP」前必须确认的承重假设，逐条对代码验证。✅ = 验过站得住，⚠️ = 验出洞、已在本方案修正。

| 假设 | 结果 | 证据 / 含义 |
|---|---|---|
| 写入富化（embedding/admission/persona/graph）谁来跑 | ✅ **设计本就支持 daemon** | `create_memory_item()` 只写主表，富化靠 `enqueue_memory_store_jobs()` 往 `background_jobs` 塞**持久化任务行**；worker 是**带租约的轮询队列**（`run_background_jobs_worker` + `reclaim_stale_running_jobs` + `lease_token`）。任何跑 worker 的进程都能捞起来处理，进程死了下次 worker 起来会 reclaim，**数据不丢** |
| services 是否有更深的 Web/请求耦合 | ✅ **干净** | grep 全 `services/` 的 `request.state`/`app.state`/`Depends`/`cookie`，只有 `voice_v2_session.py`（已暂停的语音功能，不进内核）命中，且它也只是接收 `auth_session_id` 参数 |
| 没配 embedding / sqlite-vec 没加载会不会挂 | ✅ **优雅降级** | `retrieval_primitives.py:1104` 先判 `sqlite_vec_loaded`，向量检索整段包在 `try/except`（:1140），失败回退 FTS / `LIKE`（多处 `except sqlite3.OperationalError`）。没 GEMINI key 也能搜，只是质量降；写入侧 embedding 索引失败只是 job 报错重试，主记忆已落库 |
| 是不是硬单用户 | ✅ **是**（符合预期） | 全表无 `user_id`/`tenant`/`owner_id`。这正是「你的私有本地大脑」该有的样子；但意味着 **daemon 的本地配对 token 是唯一安全边界** |
| 多进程并发写同一 `data.db` | ⚠️ **有具体隐患** | `database.py` 的 `_configure_db` 只设 `WAL`+`foreign_keys`，**没设 `busy_timeout`**。WAL 允许多读单写，但两进程各开写连接同时 commit 会**立即 `SQLITE_BUSY`**（不排队）。→ §5 D4 修正 |
| MCP 短命 stdio 进程谁跑后台 | ⚠️ **进程模型之前写错了** | 若 MCP 自带 worker 又随客户端断开就退，job 会躺表里没人及时处理。→ §3.1 / §5 D9 修正：MCP 改为**常驻后端的薄客户端**，自己不跑后台 |

**对外真实入口（适配层直接调用这些，不需重写业务）**：

- 写入：`create_memory_item()` + `enqueue_memory_store_jobs()`（两步：先落库、再入队富化）、`update_memory_item_content()`、`reprocess_memory_item()`（`services/memory_service.py`）
- 检索：`retrieve_context(query, db, *, intent=None)`（`services/retrieval_context.py:504`，混合检索主入口，**默认会调 LLM 做意图路由**，可传 `intent` 跳过）、`retrieve_graph_context(query, db)`（`services/graph_retrieval.py:70`）
- 后台：`register_memory_pipeline_job_handlers()` + 上面四个 `run_*` 循环

## 3. 目标架构：一个常驻 daemon（即现有后端）+ 薄客户端

### 3.1 关键修正：daemon 不是要「新建」的，现有后端就是

后台有 **4 个调度/worker 循环**（consolidation、weight decay、sleep agent、background jobs）。**现有的 FastAPI 后端进程已经在做这件事**：它独占共享 DB 连接（`get_shared_db`）、用 `lifespan` 把 4 个循环全挂起来、对外服务 HTTP。换句话说——

> **现有后端进程本身就是「Memory Daemon」。** 我们不需要为 MCP 新造一个引擎，只需要让 MCP 成为**这个常驻后端的薄客户端**。

这一下同时化解两个洞：
- **「MCP 短命进程谁跑后台」**：MCP 不跑后台。它把工具调用通过 loopback 转发给常驻后端，后端的 worker + 4 个循环一直在跑。MCP 进程随客户端起落都无所谓。
- **「两个写者」**：只有后端写 `data.db`。MCP（以及之后的浏览器）都只是 loopback 客户端，**永不直接持有写连接**。

```
        ┌──────────────────────────────────────────────────────────┐
        │      Memory Daemon  ==  现有 FastAPI 后端进程（常驻）       │
        │  ─ 独占 data.db 的写连接 (WAL + busy_timeout)             │
        │  ─ 跑全部后台循环: consolidation / decay / sleep / jobs   │
        │  ─ 对外: ① 现有 cookie API 给 Next.js                     │
        │         ② 新增 loopback Core API + 本地 token 给本地客户端 │
        └───────▲───────────────────▲───────────────────▲──────────┘
                │ cookie            │ loopback+token    │ loopback+token
        ┌───────┴──────┐   ┌────────┴────────┐   ┌──────┴────────────┐
        │  Next.js 前端 │   │  MCP 适配器     │   │  浏览器扩展        │
        │  (现有)      │   │  stdio 薄桥接   │   │  MV3 + content     │
        │              │   │  ← Claude/Cursor│   │  ← Chrome/Edge     │
        └──────────────┘   └─────────────────┘   └───────────────────┘
              全部是「客户端」，只有 daemon 写库、只有 daemon 跑后台
```

- **唯一写者 = daemon**：这是从 P2 就生效的**硬不变量**，不是 P4 才补的优化。
- **适配器 = 薄进程**：只做「协议翻译 + 本地鉴权 + 转发 Core API」，不开 DB 写连接、不跑后台循环。
- **独立进程 daemon 是后话**：当你希望「不开 Next.js 也能用 MCP」时，再把 daemon 拆成 headless 独立进程（§7 P2 提供 headless 启动模式；P4 才做真正的进程拆分）。在那之前，「后端进程」就是 daemon。

## 4. 目标代码结构

把现有 `backend/` 拆成「内核包 + 适配器」。建议用 monorepo / 单仓多包：

```
neo-pervault/
├── packages/
│   └── memory_core/                 # ← 抽出来的可复用内核 (无 Web 依赖)
│       ├── pyproject.toml           #    可单独 pip install / 发版
│       ├── memory_core/
│       │   ├── __init__.py          #    暴露稳定公共 API (见 §6)
│       │   ├── db/                  #    ← 原 database.py (schema/迁移/连接, 加 busy_timeout)
│       │   ├── models.py            #    ← 原 models.py (Pydantic, 非 Web)
│       │   ├── services/            #    ← 原 services/* (去掉 1 处 HTTPException)
│       │   └── runtime.py           #    新增: 统一启动/停止全部后台循环的句柄
│       └── tests/                   #    ← 原 backend/tests/* 跟随内核
│
├── apps/
│   ├── rest_host/                   # 适配器#1 = daemon 宿主: 现有 FastAPI 后端
│   │   ├── main.py                  #    ← 原 main.py, 用 MemoryRuntime, 新增 loopback Core API
│   │   ├── core_api.py              #    新增: 本地 token 的内部 Core API 路由
│   │   └── routers/                 #    ← 原 routers/*, 改 import memory_core
│   │
│   ├── mcp_host/                    # 适配器#2: MCP 记忆服务器 (薄桥接, 本批新建)
│   │   ├── server.py                #    stdio → loopback Core API; 不开 DB / 不跑后台
│   │   └── manifest / 安装脚本
│   │
│   └── browser_host/               # 适配器#3: 浏览器扩展 (薄客户端, 后续)
│       └── extension/               #    Chrome/Edge MV3 (content + background + popup)
│
├── frontend/                        # 现有 Next.js, 不变 (daemon 的 cookie 客户端)
└── docs/derivative/                 # 本系列设计文档
```

**关键约束**：
- `memory_core` **不得** import `fastapi`、`slowapi`、`routers`、`frontend` 任何东西。CI 加一条 import-lint 守住这条边界。
- 适配器**单向依赖**：`rest_host` 依赖 `memory_core`；`mcp_host` / `browser_host` 只依赖 daemon 的 loopback Core API（连 `memory_core` 都不必直接 import），更薄、更解耦。
- `models.py` 里的 Pydantic 模型保留在内核（Pydantic 不是 Web 耦合），各适配器的请求-响应 schema 放在各自适配器里。

## 5. 关键设计决策

| # | 决策点 | 选择 | 理由 |
|---|---|---|---|
| D1 | 内核与 Web 的边界 | 抽 `memory_core` 包，Web 变成适配器 | 实测耦合 ≈ 0，成本低、收益大 |
| D2 | 后台循环归属 | 只在 **daemon（现有后端进程）** 跑；任何适配器都不跑 | 避免多进程重复调度 / 抢 job |
| D3 | 唯一 HTTPException | 换成 `memory_core` 自定义领域异常（如 `MemoryError`/`AdmissionRejected`），各适配器映射成各自协议错误 | 解开唯一的 Web 依赖 |
| D4 | 多进程访问 DB | **硬不变量：只有 daemon 写库**（从 P2 生效）。适配器一律走 Core API；`_configure_db` 补 `PRAGMA busy_timeout=5000`，只读分析用 `get_db(read_only=True)` | 实测无 busy_timeout，多进程写会立即 `SQLITE_BUSY`；集中单写最稳 |
| D5 | 鉴权模型 | 现有 cookie-session 留给 Next.js；MCP / 浏览器走「loopback + 本地 token」（首次配对码换 token） | 不同宿主信任模型不同，不能共用 cookie |
| D6 | Provider 配置 | 沿用现有 env（`OPENAI_*`/`EMBEDDING_*`/`WHISPER_*`），daemon 统一加载；无 embedding 时检索自动降级到 FTS | provider 无关已做好；降级已验证 |
| D7 | 数据库位置 | 默认 `~/.pervault/data.db`（从仓库内 `backend/data.db` 迁出），所有宿主共享同一份 | 多宿主共享同一颗大脑的前提 |
| D8 | 隐私边界 | MCP/浏览器写入默认走同一套 admission 分层；加 `source` 标记（mcp/browser/voice）便于审计与按来源召回 | 复用现有准入与可观测性 |
| D9 | **MCP 进程模型** | MCP = daemon 的**薄客户端**（stdio → loopback Core API），自己不开 DB、不跑后台 | 化解「短命进程谁跑后台」与「两个写者」 |
| D10 | **检索意图路由成本** | MCP `memory_search` 默认透传 `intent` 或用便宜关键词启发式；只有显式需要时才让 daemon 调 LLM 路由 | `retrieve_context` 默认每次一次 LLM，省掉延迟/成本 |

## 6. `memory_core` 公共 API（daemon 内部只认这一层）

在 `memory_core/__init__.py` 固化一组稳定入口，封装现有 service 函数：

```python
# 写入 (内部两步: 落库 + 入队富化)
async def remember(content: str, *, tags=None, source="api", kind=None) -> MemoryItem: ...
async def update_memory(memory_id: str, content: str) -> MemoryItem: ...
async def reprocess(memory_id: str) -> dict: ...

# 检索
async def recall(query: str, *, intent=None, limit=...) -> RetrievalResult: ...   # 包 retrieve_context
async def recall_graph(query: str) -> GraphContext: ...                            # 包 retrieve_graph_context
async def stats() -> MemoryStats: ...

# 画像 / 洞察 (读)
async def get_persona() -> list[PersonaTrait]: ...
async def list_reflections(limit=...) -> list[Reflection]: ...

# 运行时 (由 daemon 宿主调用, 适配器不碰)
class MemoryRuntime:
    async def start(self): ...   # init_db + register handlers + 启动 4 个后台循环
    async def stop(self): ...    # 取消循环 + close_shared_db
```

这层是「**反腐层 / facade**」：现有 `services/*` 内部签名可继续演进，对外只保证这组 API 稳定。daemon 把它再包一层 loopback HTTP（Core API）供 MCP/浏览器调用。

### 6.1 MCP 工具面（适配器 #2 把 Core API 映射成 MCP 工具）

| MCP Tool | 映射到 | 说明 |
|---|---|---|
| `memory_store` | `remember()` | 让 Claude/Cursor 把对话里的事实/偏好写进你的私有记忆（异步富化，立即返回） |
| `memory_search` | `recall(intent=?)` | 混合检索；**可选 `intent` 参数**跳过 LLM 路由，默认走便宜启发式，必要时才 LLM |
| `memory_graph` | `recall_graph()` | 取查询相关的知识子图（实体 + 关系） |
| `memory_update` | `update_memory()` | 修订已有记忆内容 |
| `persona_get` | `get_persona()` | 注入「关于用户的稳定画像」，给模型当系统记忆 |
| `reflections_list` | `list_reflections()` | 取 sleep agent 生成的高阶洞察 |
| `memory_stats` | `stats()` | 记忆库规模/分层概览 |

> 资源（resources）方面，可把 persona + 最近 reflections 暴露成 MCP resource，让客户端每轮自动注入「长期记忆 boot context」，对齐现有 `get_boot_context` 思路。

### 6.2 浏览器扩展数据流（适配器 #3，第二条线）

```
划词/整页 → content script → background(MV3) → daemon (loopback 127.0.0.1)
   ↓ 写: POST /core/remember {content, source:"browser", url, title}
   ↑ 读: GET  /core/recall?q=...   阅读时侧边栏召回相关记忆
鉴权: 首次安装弹配对码 → daemon 校验 → 下发本地 token, 之后请求带 token
隐私: 仅 loopback、不出本机; 域名黑/白名单; 默认不抓正文全文, 只存用户主动划选
```

## 7. 分阶段改造计划

> 原则：每个 Phase 结束都「可验证、行为不破」。P0–P1 是纯重构 + 加固，P2 起才长出新产品。

| Phase | 名称 | 产出 | 验收 |
|---|---|---|---|
| **P0** | 内核抽包（行为不变） | 新建 `packages/memory_core`，把 `database.py`/`models.py`/`services/*` 迁入；去掉唯一 `HTTPException` 换领域异常；改包内 import；CI 加 import-lint 守边界 | 现有测试（迁为 `memory_core/tests`）**全绿**；前端 `tsc --noEmit` 通过 |
| **P1** | 后端坐实为 daemon + 加固 | `rest_host` 依赖 `memory_core`；`main.py` 用 `MemoryRuntime.start/stop`；`_configure_db` 加 `busy_timeout`；DB 迁 `~/.pervault/data.db`（幂等迁移 + 备份旧库 + 失败回滚） | 现有 Web/前端全链路（录音→记忆→图谱→聊天）回归通过；多进程并发写不再立即 `SQLITE_BUSY` |
| **P2** | MCP 适配器（第一条产品线） | ①daemon 加 **loopback Core API + 本地 token** 鉴权（127.0.0.1 旁路 cookie）；②daemon 加 **headless 启动模式**（不依赖 Next.js 也能常驻）；③`mcp_host/server.py` 薄桥接，透传 `intent`，**不开 DB、不跑后台** | Claude Desktop/Cursor `memory_store`+`memory_search` 闭环；**杀掉/重启 MCP 进程不丢数据**（job 由 daemon 补跑）；数据落同一 `~/.pervault/data.db` |
| **P3** | 浏览器扩展（第二条产品线） | `browser_host` MV3 扩展：划词存、阅读召回、配对鉴权，复用 P2 的 Core API + 本地 token | 网页划词存入、换页后能召回；仅 loopback |
| **P4** | 拆分独立 daemon（按需） | 把 daemon 从「兼服务 Next.js 的后端进程」拆成 headless 独立进程，REST/MCP/浏览器全部成为它的客户端 | 不开任何前端，三宿主仍可用，后台循环只跑一份 |

> 你现在确认的是这份 v2「方案」。确认后从 **P0 抽包** 真正动手——风险最低、是后面所有产品线的地基。

## 8. 风险与缓解

| 风险 | 影响 | 缓解 |
|---|---|---|
| 抽包时 import 路径大改引入回归 | 中 | 现有 200+ 测试是安全网；P0 只搬不改逻辑，逐文件改 import 并跑测试 |
| 多进程同时写 `data.db` 锁冲突 | 中高 | D4 硬不变量「只有 daemon 写库」+ `busy_timeout`；适配器一律走 Core API，不开写连接 |
| **MCP 能用的前提是 daemon 在跑** | 中 | P2 提供 headless 启动模式 + 安装脚本/开机自启；MCP 连不上时返回明确「daemon 未运行」错误 |
| MCP/浏览器写入污染记忆质量 | 中 | 复用现有 admission 分层 + `source` 标记，可按来源过滤/审计 |
| 没配 embedding 检索质量降 | 低 | 已验证自动降级 FTS/`LIKE`；安装引导提示配 `GEMINI_API_KEY` 解锁向量召回 |
| 本地 token 泄露 = 全部记忆暴露（单用户无隔离） | 中高 | token 存 `~/.pervault` 0600 权限；仅 loopback 绑定；配对码一次性、可吊销 |
| `~/.pervault` 迁移丢数据 | 高 | P1 迁移幂等 + 备份旧库 + 失败回滚 |
| 隐私（浏览器抓网页） | 高 | 仅 loopback、默认只存用户主动划选、域名黑白名单、明确不上云 |

## 9. 验收标准（本方案落地后整体）

- `memory_core` 可被 `pip install` 后在**不装 fastapi** 的环境里 `import` 并跑通 `remember()` / `recall()`。
- import-lint：`memory_core` 包内 0 处 `fastapi`/`routers`/前端引用。
- 现有 Web + 前端主链路回归全绿。
- **唯一写者**：运行期只有 daemon 进程持有 `data.db` 写连接（可用 `lsof` 验证）。
- Claude Desktop（或 Cursor）通过 MCP 完成「存一条 → 搜回来」闭环；**MCP 进程重启不丢富化结果**。
- 浏览器扩展完成「划词存 → 换页召回」闭环，全程 loopback。
- 三个宿主共享同一份 `~/.pervault/data.db`，记忆互通。

## 11. 变更记录 v1 → v2（4 个洞怎么补的）

| 洞 | v1 的问题 | v2 的修正 | 落点 |
|---|---|---|---|
| **洞1 进程模型** | MVP 让 MCP「进程内加载内核 + 自跑后台」，短命 stdio 进程会让富化 job 滞留 | 否掉该退化方案；**现有后端即 daemon，MCP 改薄客户端**，自己不跑后台 | §3.1、D9、P2、§10 |
| **洞2 并发写** | 断言「WAL 能扛多进程写」，实际 `_configure_db` 无 `busy_timeout`，会立即 `SQLITE_BUSY` | 定**硬不变量「只有 daemon 写库」**（P2 生效，非 P4）+ 补 `busy_timeout` | D4、P1、§8 |
| **洞3 两个写者** | MCP daemon + 现有 Web 后端 = 两个独立写进程 | MCP/浏览器都是 daemon 的 loopback 客户端，**永不直接写库**；现有后端就是那个唯一 daemon | §3.1、D2、D4 |
| **洞4 检索成本** | `memory_search` 默认每次一次 LLM 意图路由 | `recall`/工具面支持**透传 `intent` + 便宜启发式兜底** | D10、§6.1 |

> 另记录第二轮**验过没问题**的点（增强信心，非洞）：持久化租约 job 队列（进程死不丢数据）、services 零 Web 耦合、无 embedding 优雅降级、单用户本地设计。详见 §2.2。
